# Lesson 4: типизация Supabase и страница сообщений

## План выполненной работы

1. Подключить автоматическую типизацию Supabase.
2. Исправить запросы к сообщениям под актуальную схему базы данных.
3. Переделать страницу сообщений в интерфейс чата.
4. Разделить большую страницу `page.tsx` на отдельные компоненты.
5. Заменить email в шапке на имя, фамилию и должность менеджера.
6. Добавить отображение статусов клиента и назначения менеджера.
7. Добавить форму назначения клиента менеджеру.
8. Заблокировать ответ клиенту, если диалог не назначен текущему менеджеру.
9. Сохранять черновики ответов отдельно для каждого клиента.
10. Проверить проект через TypeScript, lint и production build.

## 1. Автоматическая типизация Supabase

Подключили автоматическую генерацию типов Supabase для проекта.

Добавили команду в `package.json`:

```json
"supabase:types": "supabase gen types typescript --linked --schema public > lib/supabase/database.types.ts"
```

Теперь типы можно обновлять командой:

```bash
pnpm supabase:types
```

Сгенерированный файл:

```txt
lib/supabase/database.types.ts
```

Подключили тип `Database` к Supabase-клиентам в:

```txt
lib/supabase/server.ts
```

Теперь `createSupabaseServerClient()` и `createSupabaseAdminClient()` используют типизированный Supabase client.

## 2. Исправление запросов под новую схему БД

После подключения типов стало видно, что таблица `messages` больше не содержит поля клиента напрямую:

```txt
chat_id
user_id
username
first_name
last_name
```

Теперь сообщения связаны с клиентом через `client_id`, поэтому запросы были исправлены:

- сообщения берутся из `messages`;
- данные клиента берутся из связанной таблицы `clients`;
- сообщения фильтруются по выбранному клиенту.

Основные функции:

```txt
getClients()
getMessagesByClientId(clientId)
getManagers()
getAuthorizedManager()
```

Файл:

```txt
app/messages/queries.ts
```

## 3. Переделка страницы сообщений

Страница сообщений была переделана из обычного списка в интерфейс, похожий на чат.

Что сделали:

- слева отображается список всех клиентов;
- справа отображается диалог выбранного клиента;
- выбранный клиент хранится в URL через query-параметр:

```txt
/messages?client=<client-id>
```

- если клиент не выбран, автоматически выбирается первый клиент;
- сообщения показываются только для выбранного клиента;
- сообщения сгруппированы по дням;
- внутри одного дня сообщения находятся на общем белом фоне;
- текст сообщения начинается слева;
- время сообщения отображается справа;
- высота чата фиксированная, поэтому при переключении клиентов интерфейс не прыгает.

## 4. Разделение страницы на компоненты

Чтобы `page.tsx` не был перегружен JSX-разметкой, страницу разделили на компоненты.

Новые файлы:

```txt
app/messages/clients-sidebar.tsx
app/messages/chat-panel.tsx
app/messages/message-list.tsx
app/messages/reply-composer.tsx
app/messages/assignment-form.tsx
```

Ответственность компонентов:

- `clients-sidebar.tsx` - список клиентов слева;
- `chat-panel.tsx` - шапка выбранного диалога, сообщения и поле ответа;
- `message-list.tsx` - группировка сообщений по дням;
- `reply-composer.tsx` - поле для ответа менеджера;
- `assignment-form.tsx` - выбор менеджера для назначения клиента.

Теперь `app/messages/page.tsx` в основном отвечает за загрузку данных и сборку страницы.

## 5. Шапка с данными менеджера

В шапке страницы больше не показывается email авторизованного пользователя.

Теперь показывается:

- имя менеджера;
- фамилия менеджера;
- должность менеджера.

Логика:

1. Получаем текущего авторизованного пользователя Supabase.
2. По `user.id` ищем запись в таблице `managers`.
3. В шапку передаем:

```ts
manager.name
manager.surname
manager.position
```

Файлы:

```txt
app/messages/header.tsx
app/messages/page.tsx
app/messages/queries.ts
```

## 6. Назначение клиента менеджеру

Добавили возможность назначать выбранный диалог любому менеджеру из системы.

Для этого используется таблица:

```txt
client_assignments
```

В списке клиентов теперь есть метки:

- `Not resolved` - статус задачи по клиенту;
- имя назначенного менеджера;
- `No assign`, если клиент никому не назначен.

Важно: в текущей схеме Supabase нет отдельного поля для статуса resolved/not resolved, поэтому `Not resolved` пока отображается как визуальная метка. Чтобы сделать это настоящей логикой, нужно добавить поле статуса в базу данных.

Назначение менеджера происходит через форму в шапке диалога.

При отправке формы вызывается:

```ts
assignClientAction()
```

Action записывает данные в `client_assignments`:

```ts
{
  client_id: clientId,
  current_manager_id: managerId,
  assigned_by_manager_id: currentManager.id,
  updated_at: new Date().toISOString(),
}
```

Используется `upsert` по `client_id`, поэтому повторное назначение обновляет текущую запись, а не создает дубликаты.

## 7. Ограничение ответа по назначению

Добавили правило: отвечать клиенту может только назначенный менеджер.

Логика:

- если клиент никому не назначен, textarea и кнопка `Send` заблокированы;
- если клиент назначен текущему менеджеру, ответ разрешен;
- если клиент назначен другому менеджеру, ответ заблокирован;
- чтобы ответить, менеджер должен переназначить клиента на себя.

Причина блокировки отображается над полем ответа.

Файлы:

```txt
app/messages/chat-panel.tsx
app/messages/reply-composer.tsx
```

## 8. Черновики ответов для каждого клиента

Сделали так, чтобы текст в поле ответа сохранялся отдельно для каждого клиента.

Поведение:

- если написать текст в одном диалоге и перейти в другой, текст не переносится;
- у другого клиента поле ответа будет пустым или со своим черновиком;
- если вернуться к первому клиенту, введенный ранее текст восстановится;
- если очистить поле, черновик для этого клиента удаляется.

Черновики хранятся в `localStorage` по ключу:

```ts
messages:reply-draft:<client-id>
```

Для этого `reply-composer.tsx` стал client component.
